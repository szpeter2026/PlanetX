## 三、关键缺失要素（逐层分析）

### 🔴 缺失层 1：PlanetX C端完整产品 — **缺失度：85%**

当前仅有品牌概念页（九宫格），缺乏可交互的完整产品。

| 缺失组件 | 优先级 | 说明 | 后端需求 |
|----------|--------|------|----------|
| **星际人格测试** | P0 | PlanetX 九宫格中「行为钩子」节点明确提到"扫码解锁星际人格测试"。这是用户钩子/裂变入口 | 需要 `/v1/personality/*` 或复用 MBTI 测评 |
| **个人主页/仪表盘** | P0 | 求职者的"星球"，展示匹配进度、技能图谱、求职状态 | 需要 `/v1/dashboard/*` 聚合接口 |
| **组队建舱（社交裂变）** | P1 | 九宫格中「社交裂变」节点："3人成团，解锁隐藏星图" | 需要 `/v1/team/*` 组队、邀请、解锁逻辑 |
| **情绪陪伴功能** | P1 | 九宫格中「情绪价值」节点："允许你信号不良——静音即自由" | 部分已有（诗词推荐），需扩展到情绪识别/白噪音/树洞 |
| **求职进度追踪** | P1 | 投递记录、面试日程、offer 对比 | 需要 `/v1/tracker/*` |
| **技能成长路径** | P2 | 基于匹配结果推荐学习路径 | 需要 `/v1/growth/*` |
| **PlanetX 小程序** | P0 | 完整可交互的 C 端小程序（当前仅有迁移方案） | 前端工作 |

**当前关联：** looma-zervi 已有 `/v1/ask`（问答）、`/v1/jobs/match`（匹配）、MBTI 测评。这些是 PlanetX 的基础功能，但缺少用户体系、社交、成长等模块。

---

### 🔴 缺失层 2：T空间 B端完整产品 — **缺失度：90%**

当前仅有产品海报（营销物料），技术层面严重不足。

| 缺失组件 | 优先级 | 说明 | 后端需求 |
|----------|--------|------|----------|
| **企业入驻/注册** | P0 | 企业 HR 创建账号、配置企业信息 | 需要 `/v1/enterprise/*` 多租户体系 |
| **AI 模拟面试官** | P0 | T空间海报四大核心能力之一："24h陪练，根据岗位JD自动生成面试题" | **全新模块**：`/v1/interview/*`（题目生成、实时对话、评分反馈） |
| **企业知识库 RAG** | P0 | 海报四大核心能力之一："员工手册、产品文档导入，新人直接问 AI" | 部分已有（文档导入管道），需补齐企业级知识库管理面板 |
| **JD 管理与发布** | P0 | 企业 HR 创建/编辑/发布职位描述 | 需要 `/v1/jobs/*` CRUD（当前仅 match，无管理） |
| **候选人看板** | P1 | 匹配结果可视化、候选人阶段管理、面试安排 | 需要 `/v1/pipeline/*` 招聘管道 |
| **企业 Web 管理面板** | P0 | HR 日常操作界面（非技术预览，完整产品） | **全新前端项目** |
| **多租户数据隔离** | P0 | 企业 A 的数据与企业 B 完全隔离 | 需要在 looma-zervi 中引入租户体系 |
| **企业分析报告** | P1 | 招聘效率、渠道分析、团队匹配度 | 部分已有（report_gen.py），需接口化 |
| **内测申请流程** | P0 | 海报中"前30家永久免费"的落地流程 | 需要 `/v1/beta/*` 申请/审批/开通 |

---

### 🟡 缺失层 3：双向数据管道（C端 ↔ B端）— **缺失度：100%**

这是 PlanetX-T空间 生态的核心飞轮，当前完全缺失。

| 缺失组件 | 优先级 | 说明 |
|----------|--------|------|
| **统一用户身份** | P0 | PlanetX 求职者 = 个人用户，T空间 = 企业HR用户，需要统一认证体系区分角色 |
| **求职者 → 企业匹配管道** | P0 | PlanetX 用户的简历/技能数据 → 自动匹配 T空间 企业发布的职位 |
| **企业 → 求职者触达** | P1 | T空间 企业可主动搜索/邀请 PlanetX 上的求职者 |
| **数据脱敏与隐私** | P0 | C端用户数据在流向B端前必须脱敏（PlanetX 的"隐私优先"原则） |
| **双向评价体系** | P2 | 求职者对企业的评价 + 企业对候选人的反馈 |

---

## 五、核心待决策问题

| # | 问题 | 选项 | 影响 |
|---|------|------|------|
| 1 | **多租户方案** | A. PostgreSQL schema 隔离 / B. 单表+tenant_id | A 安全性高但运维复杂，B 简单但需严格权限控制 |
| 2 | **PlanetX 小程序 vs T空间小程序** | A. 合并在一个 / B. 两个独立小程序 | A 需做角色切换，B 维护两套但体验独立 |
| 3 | **AI 模拟面试的实现方式** | A. 复用 LiteLLM + 提示工程 / B. 独立微服务 | A 快速上线，B 更灵活但增加运维 |
| 4 | **真实职位数据对接** | `api.yaml` 已规划"V1+ 接入真实 API（BOSS 直聘等）" | 需要爬虫/API 对接 + 合规审查 |
| 5 | **企业知识库的向量隔离** | A. 每个企业独立 collection / B. 共享 pgvector + tenant_id 过滤 | A 物理隔离，B 更省资源 |

### 最关键的 3 个缺失要素（立即需要补齐）

1. **PlanetX 微信小程序**（C端唯一可交互入口）—— 迁移方案已有，代码未写

### 现阶段的正确路径

```
当前 → looma-zervi 部署 HTTPS 域名
    → PlanetX 小程序代码落地
    → 企业入驻 + JD 管理 API
    → T空间 Web 管理台 MVP
    → 双向匹配管道打通
    → AI 模拟面试
    → 社交裂变 + 情绪陪伴
```

> **关键洞察：** looma-zervi 的 P1 底座完成是最大优势——它提供了统一的技术地基。下一步不是继续打磨后端，而是**快速补齐前端产品矩阵**（小程序 + Web管理台），让用户/企业能真正用起来，再回填缺失的API模块。
